Distributed Systems

Introduction

Raft is a distributed consensus algorithm, which was posed as an easier to understand alternative to Paxos. The Raft consensus algorithm works on replication of state machines, which is in itself broken down into three separate but concomitant tasks:

  1. Leader Election
  2. Log Replication
  3. Safety

In addition to these, cluster membership changes are also accounted for in the original paper. In this blog post, we will go through the process of analyzing each of these avenues, and writing a very simple replicated state machine in Python with a custom implementation of Raft and run it across some docker containers.

On Raft

This section explains the original raft paper in its most essential parts. Feel free to skip this portion if you already know what raft is and how it works. Or stick around for an introduction and a brief refresher 😄

About Replicated State machines:

Replication in distributed systems occurs usually through one of three ways:

  1. Transactional Replication used in transactional systems such as distributed databases
  2. State machine replication where

Type of replication:

There are two ways replication occurs:

  1. Active Replication
  2. Passive Replication - Raft is this type of replication where a master server forwards data to all other nodes in the cluster

Raft keywords:

Some keywords should be defined before we describe the entire algorithm:

  1. Client: External machines that communicate with the cluster.
  2. Role: Current role of the server in the cluster. This can be one of Leader, follower or Candidate
    • Leader: All changes by clients are sent to this node in the cluster
    • Follower: Followers listen for commands from Leader
    • Candidate: When followers do not receive any heartbeat message from the leader, they default to a Candidate state, showing their interest in becoming the leader
  3. Term: Time in raft is divided into discrete terms. Each term has at most one leader and may have no leaders at all (in case of a split vote). The Term is a monotonically increasing counter that is tracked by every node in the cluster.
  4. Log: Log is the array of changes that has been made to the internal state machine. Log is unique to each node in the cluster and for all agreeing and correct nodes, Log should be similar. One example could look like this: SET x 10 -> SET y 10 -> DEL x -> SET y 30.
  5. Commit and Apply: The logs are committed and applied to the internal state machine of the servers. To commit a log means that the log has been stored in the node successfully, and that can be applied to the internal state machine of the node later on.
  6. Election Timeout: Time for which a follower node waits to receive a heartbeat message or election votes. When this timeout expires, the election starts / gets canceled.
  7. Split vote: No node in the cluster could get a majority vote to become the leader.

Now that we have some idea of Raft under our belt, let’s analyze the two main parts of Raft in conjunction with the third - Leader election and Log Replication and their safety conditions and how they are guaranteed by Raft.

The Leader election:

Every node in the cluster begins in a follower state with a certain timeout. When this timeout expires and no heartbeat or vote request is received from any of the other nodes in the cluster, the follower node increases its monotonic term counter and changes its state to a candidate. Then it sends out vote request to all other nodes in the cluster. The Algorithm can be enumerated as follows:

  1. Start as follower
  2. When follower:
    1. If no heartbeat received until election_timeout time, increase own term and mark oneself as candidate. Then send voterequestRPC to all known nodes in cluster
    2. If heartbeat received:
      1. If heartbeat is greater than or equal to own term, acknowledge the sender node as the leader
  3. When Candidate:
    1. Send out voterequestRPC to all nodes in the cluster and wait until election_timeout. When waiting:
      1. If voterequestRPC received or heartbeat received whose term is equal to or greater than own term, then acknowledge sending node as leader.
      2. If voterequestRPC received from a node whose term is less than own term, reject that RPC.
      3. If the node does not receive majority votes even after election_timeout time, it will increment its own term, and start another election cycle by sending out voterequestRPC. However, an issue here might be that if the servers wait for the same duration as they did before, they will have a high chance of having the same outcome as they did before, i.e. a split vote. In order to mitigate this the election_timeout is randomized with a small variation, ( 150-300ms in paper ). This ensures that split votes do not persist.
  4. When leader:
    1. Send out periodic heartbeats to all nodes in the cluster. In order to decrease the chance that the follower might initiate an election before receiving the heartbeat, send out multiple heartbeats per election_timeout time so that this does not happen (Visualization for this is well-illustrated in raft website)
