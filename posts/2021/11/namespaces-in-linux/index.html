Namespaces in Linux

Namespaces decide what subset of resources the processes running on a system can access or “see”. With namespaces, we can have isolated mountpoints, PID spaces (so that two processes on the OS can think their PID is the same), hostnames and network interfaces isolated to processes. These resources may have a different mapping on the host system. For an example, the first process running inside a PID namespace might think that its PID is 1 but in the host system, it is actually mapped to some other PID, like 1234.

The list of running namespaces in the host system can be extracted by running the lsns command. Namespaces provide the basis of isolation between processes running on the operating system and allocateable resources. There are various types of namespaces in linux. Some of the namespaces being used to build BeeJudge1 are explained briefly below:

PID Namespaces

PID namespaces allow the isolation of process identifiers (PIDs) between different namespaces. As such, two processes running on the host system might think that they have the same PID when in reality they are different. Newly created namespaces assign PID of 1 to the first run process inside the namespace and subsequent PIDs to the processes after that. For a simple demonstration, let’s look at the output after creating a new PID namespace with bash:

Create a new PID namespace with the unshare command:

$ sudo unshare -p /bin/sh

On the host system:

$ ps -ax | grep /bin/sh
1018662 pts/2    S      0:00 sudo unshare -pf /bin/sh
1018663 pts/2    S      0:00 unshare -pf /bin/sh
1018664 pts/2    S+     0:00 /bin/sh

Inside the namespace:

# echo $$
1

Here we can see that although the sh process has PID of 1018664 on the host system, it has a PID of 1 inside the new namespace we created. We can also see that the PID namespaces are different by running the following commands on the system:

On the host system:

$ sudo ls -lah /proc/1018664/ns
total 0

Mount Namespaces

If you tried the above command in your system and ran a ps -ef command, you would be surprised to see that all the processes running on the host system are enumerated in the output. Technically it seems like there was no isolation of PIDs whatsoever! This happens due to a reason. When we created a new namespace, the old mountpoints are copied along to the new namespace. And commands like ps work on /proc mount to extract information about running processes.

In order to avoid this complication, we will depend on mount namespaces. First of all, we will create a new PID namespace along with a mount namespace and fork a /bin/sh inside it as usual.

Create the namespace on host machine

$ mkdir /tmp/namespaceroot
$ unshare -pmf /bin/sh

Why we created a namespaceroot folder will be evident shortly. Now inside the namespace, we run a series of commands to mount a temporary filesystem

Inside the namespace

# mount -t tmpfs tmpfs /tmp/namespaceroot
# mkdir -p /tmp/namespaceroot/proc
# mount -t proc proc /tmp/namespaceroot/proc
# ps -ef

Along with proc, we will also isolate the /tmp mount

On the host

$ mkdir -p /tmp/newtmp

Inside the namespace

# mount -o bind,nodev /tmp/newtmp /tmp
# ls /tmp

Now if you create any file inside the /tmp/newtmp folder in host system, it should be readable inside the namespace.
