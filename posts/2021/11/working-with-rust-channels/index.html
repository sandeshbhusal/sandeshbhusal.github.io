Working With Rust Channels

By | November 19, 2021 | 2 minutes
rust threads

Channels are an abstraction for sharing data between concurrently-running threads. For an example, you have a mapreduce system, and map and reduce threads are running parallelly. Each mapped output needs to be reduced using a reducer thread. Instead of passing data to a shared memory, which would require locks, you could just pass data to a channel and it would be received by the corresponding receiver thread. Using [[Go]]’s language documentation,

Do not communicate by sharing memory; instead, share memory by communicating.

Example: Implement a chat system or a system where many threads perform parts of a calculation and send the parts to one thread that aggregates the results. Channels are supplied by std::sync, and a type of channel called mpsc is used in the book, which stands for multiple producer single consumer.

use std::sync::mpsc;
use std::thread

fn main() {
  let (tx, rx) = mpsc::channel(); 
  thread::spawn(move || {
    let val = String::from("hi"); 
    tx.send(val).unwrap(); 
  }); 
  let received = rx.recv().unwrap(); 
  println!("Got: {}", received);
}

Code Synopsis:

  • thread::spawn used to spawn new thread
  • move used so that the new thread owns the tx object
  • tx.send needs to be unwrap()ped because receiving end might’ve been dropped, so we might have errors.
  • recv() call does not block, instead returns Result<T, E>, so that if no data is available, it returns an ‘Error’.

    enum Message{
  JoinRequest(String),
  DataRequest(String)
}

let (transmitter, receiver) = mpsc::channel();
thread::spawn(move || {
  let x = Message::DataRequest(String::from("I want data"));
  let y = Message::JoinRequest(String::from("I want to join!"));
  transmitter.send(x).unwrap();
  thread::sleep(time::Duration::from_secs(2));
  transmitter.send(y).unwrap();
});

for item in receiver{
  match item {
    Message::JoinRequest(join) => {
      println!("I got {}", join);
    },
    Message::DataRequest(data) => {
      println!("I got {}", data);
    }
  }
}
    • In the same way, data can be “tagged” while passing to a channel with enums. This is a simple example, which can be used in cases like, when we have peers in a network that listen on a single port for both “join” request of new nodes, and “sync” requests to request for data. However, using enums is not really a good idea, since Rust allocates for the largest size in receiver. Instead, we could use Box to point to different types of data that can be available. Or Arc if it needs to be passed to multiple threads.

A variable cannot be “reused” after being sent down the channel, because the receiving thread might drop it or change it. So a third println!("I sent {:?} down the channel", y); would fail in the above code.
